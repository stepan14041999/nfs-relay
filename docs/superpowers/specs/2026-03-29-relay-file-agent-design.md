# Relay File Agent — Design Spec

## Overview

Three-component system for accessing a Windows machine's filesystem from another Windows machine through a relay server. Agent exposes files, relay forwards packets, mounter presents them as a local drive.

## Architecture

```
[Agent (Win Client 1)] ←—TLS 1.3—→ [Relay Server] ←—TLS 1.3—→ [Mounter (Win Client 2)]
     AGENT_ROOT                      forwards NDJSON                  mounts Z:\
```

## Protocol

- **Transport:** TLS 1.3 with mTLS. Both clients present certificates signed by a shared CA.
- **Ciphers:** `TLS_AES_256_GCM_SHA384:TLS_CHACHA20_POLY1305_SHA256`
- **Framing:** NDJSON (newline-delimited JSON), parsed via Node.js `readline`.
- **Handshake:** First message identifies role:
  - Agent: `{"role":"agent","id":"pc1"}`
  - Mounter: `{"role":"mounter","agentId":"pc1"}`
- **Request/Response:** Every request carries a numeric `id`. The response echoes the same `id`.
- **Operations:**
  - `readdir` → `string[]`
  - `stat` → `{size, mode, mtime, atime, nlink, uid, gid}`
  - `read` → `{data: base64, pos, len}`

## Components

### relay-server.js (Linux/any)

- TLS 1.3 server with `requestCert: true, rejectUnauthorized: true`
- Maintains `Map<id, socket>` for agents and `Map<agentId, socket>` for mounters
- Parses only the first message (handshake) to determine role and register
- Forwards all subsequent packets from mounter to matched agent and vice versa
- On disconnect of either side, sends `{"type":"disconnect"}` to the other
- Listens on `RELAY_PORT` (default `8443`)

### agent.js (Windows Client 1)

- Connects to relay server via TLS, sends handshake `{"role":"agent","id":"<AGENT_ID>"}`
- Handles three operations against local filesystem:
  - `readdir`: `fs.readdir` on resolved path
  - `stat`: `fs.stat` → returns `{size, mode, mtime, atime, nlink, uid, gid}`
  - `read`: `fs.open` + `fs.read` at given position/length → base64 encoded data
- `AGENT_ROOT` (env var, default `C:\Users\Stepa`): all paths resolved relative to this
- Path traversal protection: `path.resolve(ROOT, requestedPath)` must start with ROOT
- Auto-reconnects to relay every 5 seconds on disconnect
- Sends error responses with `{id, error: "message"}` on failures

### mounter.js (Windows Client 2)

- Connects to relay server via TLS, sends handshake `{"role":"mounter","agentId":"<AGENT_ID>"}`
- Mounts `MOUNT_POINT` (default `Z:\`) via `fuse-bindings` + WinFsp
- Translates FUSE callbacks into relay requests:
  - `getattr` → `stat`
  - `readdir` → `readdir`
  - `open` → `stat` (validate existence)
  - `read` → `read`
- Pending requests tracked in `Map<numericId, {resolve, reject}>` or callback-based
- On relay disconnect: unmount drive, then reconnect after 5 seconds
- Assigns monotonically incrementing numeric IDs to outgoing requests

## Configuration

All via environment variables:

| Variable | Default | Used by |
|---|---|---|
| `RELAY_HOST` | `localhost` | agent, mounter |
| `RELAY_PORT` | `8443` | all three |
| `AGENT_ROOT` | `C:\Users\Stepa` | agent |
| `AGENT_ID` | `pc1` | agent, mounter |
| `MOUNT_POINT` | `Z:\` | mounter |

## Security

### Certificate Generation (`gen-certs.sh`)

Generates a self-signed CA, then signs:
- `server.crt` / `server.key` — for relay-server
- `client1.crt` / `client1.key` — for agent
- `client2.crt` / `client2.key` — for mounter

All placed in `certs/` directory.

### Path Traversal Protection (agent)

Every requested path is resolved against `AGENT_ROOT` using `path.resolve`. The result must begin with `AGENT_ROOT`; otherwise the request is rejected with an error.


## Prerequisites

- Node.js 18+
- WinFsp installed from https://winfsp.dev (for mounter)
- OpenSSL (for gen-certs.sh)

## Error Handling

- Agent returns `{id, error: "message"}` for any fs operation failure
- Mounter maps error responses to appropriate FUSE error codes (ENOENT, EIO, etc.)
- Relay sends `{"type":"disconnect"}` when counterpart disconnects
- Both clients auto-reconnect with 5-second backoff
